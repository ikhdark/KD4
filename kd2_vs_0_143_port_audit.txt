  broad delete-heavy rollbacks as evidence, not as direct port candidates.
- KD4 had a pre-existing tracked dirty file before this audit:

Decision labels
---------------
PORT:
  Worth bringing forward, but still needs normal KD4 call-path validation.

MAYBE:
  Useful idea or partial improvement, but current v0.143.0 may already have
  newer behavior, or the change needs deeper current-fork reconciliation.

SKIP:
  Do not port wholesale. Usually generated drift, an older rollback, or already
  present/superseded upstream behavior.

  skills/src/assets/samples/*

Reason:
  Mostly README breadcrumbs and reductions to plugin-creator samples/validators.
  No clear better runtime behavior.

Suggested port order
--------------------
1. Small low-risk user-facing wins:
   /diff untracked cap, /ideas command, selected README/AGENTS/SOURCE_MAP notes.

2. Local fork proof chain:
   build_info, local runtime provenance, app-server initialize metadata, CLI
   doctor/runtime output.

3. Network/client support:
   custom CA transport and only the required HTTP/WebSocket call-site wiring.

4. Local validation discipline:
   source tools, verify_local scripts/tool, shell output summary, session evidence
   memory, dirty-worktree baselines.

5. Command execution hardening:
   command preflight, command shape, PowerShell script-body handling, rg linting.

6. Config diagnostics:
   config explain, mcp requirements, permission tests, coding budget constants.

7. App-server protocol/runtime improvements:
   startup fallback, structured turn-start validation, client quirks, targeted
   protocol fields with generated schema regeneration.

8. Persistence/runtime targeted fixes:
   state indexes, rollout read repair, rollout search snippets, memory atomic
   writes, cloud task override propagation, app-server-test-client Windows helper.

9. Remote-control enrollment:
   port only after a focused test plan because it is runtime/security-sensitive.

Validation requirements for actual port work
--------------------------------------------
- Rust crate/source changes: run focused cargo/just checks for touched crates.
- Protocol or app-server schema changes: regenerate schema through the owning
  recipe and run focused app-server protocol tests.
- Config schema changes: regenerate config schema through the owning recipe.
- Safety/exec/sandbox/apply-patch changes: run focused accepted/rejected path
  tests and do not change unrelated approval/sandbox semantics.
- Desktop-visible changes: rebuild/publish local codex, restart Desktop, and
  prove the runtime chain with current process path, binary hash/version, and
  app-server initialize metadata.

Bottom line
-----------
KD2 contains real local improvements worth preserving, especially local runtime
proof, custom CA support, source/verification tooling, command preflight,
config diagnostics, memory/evidence discipline, and selected app-server fixes.

KD2 also contains many older rollbacks relative to v0.143.0. Do not copy KD2
wholesale. Port narrowly by feature slice, keep v0.143.0's newer behavior where
it is richer, and regenerate generated outputs only from source changes.
